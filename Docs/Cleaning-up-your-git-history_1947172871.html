Blast Royale : Cleaning up your git history

Created by Miha Krajnc, last modified on May 25, 2022

In a big project git history is quite important, so we should take some care into ensuring that our commits are properly structured. While we can all strive to do it as we go, there‚Äôs bound to be some mistakes and being comfortable with modifying your git history is a very useful skill to have.

Initial setup

You technically don‚Äôt need anything specific to do an interactive rebase, however git uses vim as the editor by default, which, most (normal) developers probably aren‚Äôt that familiar with. So we‚Äôll replace it's default editor with something more appropriate. You‚Äôll be doing all of this in a terminal, so open it up.

Note: This assumes you have Homebrew installed. If you don‚Äôt just run:

Interactive rebase tool

https://github.com/MitMaro/git-interactive-rebase-tool

This is a tool that you can hook into git to help you setup your interactive rebase. If this is not installed / setup, git will default to vim (which is perfectly fine to use, if you know how to).

Install:
brew install git-interactive-rebase-tool
Setup:
git config --global sequence.editor interactive-rebase-tool

Git text editor

On top of a ‚Äúrebase tool‚Äù we also need to set up a text editor (e.g. for when you want to rename your commits), otherwise git will default to vim again.

This depends on your own preference, and which text editor you use. The simplest solution is to use Nano, a simple terminal based editor (very much not like vim), which comes preinstalled on mac:

git config --global core.editor "nano"

A better solution is to set it up with a regular GUI based text editor, like:

Sublime Text
git config --global core.editor "subl -n -w"
Visual Studio Code
git config --global core.editor "code --wait"
Atom
git config --global core.editor "atom --wait"

You can get more info here.

Doing an interactive rebase

Great, now you have everything setup and are ready to do your first interactive rebase. Take a Xanax or two, grab a pint, and let‚Äôs get to it, the first one is gonna be stressful.

Keep in mind that an interactive rebase rewrites your history, and requires you to force push your branch to BitBucket. Make sure there aren‚Äôt any changes on the remote branch, or they could get lost (although git will produce an error if there are when you try to force push).

Before we start

If you just want to change the message of your last commit, you can do:

git commit --amend

Doing the deed

  1. Make sure you are up to date with your remote branch.

  2. Open your terminal and go to the folder of the project.

  3. Check how many commits in your branch you want to change. I.E. How many commits from the latest one you want to go back.

  4. Start the interactive rebase:
    Option 1:

    git rebase -i HEAD~5

    In this case, HEAD~5 means you want to go 5 commits back from your latest commit. Make sure you don‚Äôt go back more than what‚Äôs on your branch (i.e. if your branch only has 3 commits, only go back 3 commits).

    Option 2:

    git rebase -i d5193fb20f4d23f6dff9ece7d19fee9c27e5c400

    Works the same way as Option 1, but instead of telling git you want to go back X number of commits, you tell it to go back to a specific commit.

  5. The interactive rebase tool will now kick in:

    This window allows you to reorder your commits, delete them, merge them with the previous one etc‚Ä¶ You can press ? on your keyboard to see all the possible commands. Upper case letters are done with shift as you would normally. The most important ones are:

    1. Up and Down arrows to move your selection.

    2. ‚Äúr‚Äù to change the commit message (reword).

    3. ‚ÄúW‚Äù to confirm your changes and exit / continue.

    4. ‚ÄúQ‚Äù to exit without doing anything.

    5. You can also move commits up and down, remove them, merge them, change their content etc‚Ä¶ Check the help section to learn more, or read up on the rebase command here.

  6. To change the commit message, go to that commit(s) and click ‚Äúr‚Äù. After you marked all the commits you want to change, click W (Shift + w) to save and exit.

  7. If you chose commits to be reworded, git will now open your text editor (that you set up above) and give you the option to do that:

    Now just change the message at the top (ignore the lines starting with ‚Äú#‚Äù), and close the window.

  8. When you close the window / exit out of your text editor, git will detect that and, if you have more commits to reword, will open your text editor for the next one, until there aren‚Äôt any left.

That‚Äôs it, you‚Äôve done your first interactive rebase! It‚Äôs a very powerful tool, but if you only use it for rewording commits you‚Äôll be fine.

The last thing to do is push your changes to the remote branch. Since we changed the git history, the local and remote branches have diverged (commits will get different hashes when changing their text, so git thinks they‚Äôre different commits). You will need to do a force push, which will overwrite the remote branch with your local branch:

Note: The --force-with-lease option does some additional checks to make sure you‚Äôre not rewriting something you shouldn‚Äôt be, like commits from other people that weren‚Äôt on your local branch. If you use --force instead, that will skip those checks and immediately rewrite the remote branch.

TL;DR;

  1. In the terminal go to your project.

  2. Run git rebase -i HEAD~5 (replace 5 with the number of commits you want to go back from your latest one)

  3. Select the commits to reword with Up and Down arrows, and press ‚Äúr‚Äù on them.

  4. Press Shift+W to save and exit.

  5. Do your changes in the text editor and close it.

  6. Run git push --force-with-lease

NOTE: If you just want to change the message of your last commit you can run git commit --amend and change the message in the text editor, as you would with an interactive rebase.
