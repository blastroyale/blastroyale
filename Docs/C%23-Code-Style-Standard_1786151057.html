Naming

Files

Name the files according to their class names. For partial classes, prefix the file name with the class name and suffix it with the partial class purpose.

// Located in Task.PlayerTest.cs
public partial class Task
{
}

// Located in Task.EnemyTest.cs
public partial class Task
{
}

Namespaces

Namespaces need to be defined for every class, enum, interface or struct. They should match the project source folder structure.

// CORRECT
namespace Game.Components.World
{
	public class MyClass
	{
	}
}

// WRONG
public class MyClass
{
}

Access Modifiers

Fields, classes, properties and methods access modifiers have to be declared and should always precede over modifiers

// CORRECT
public class MyClass
{
	public int field;
	public static int field;

	public int Property { get; set; }
	
	private void Method()
	{
	}
}

// WRONG
class MyClass
{
	int field;
	static public field;

	int Property { get; set; }
		
	void Method()
	{
	}
}

Built-In Types

Please prefer the built-in types over their according .NET Framework counter-part

// CORRECT
string text = "Hello World";
int index = 0;

// WRONG
String text = "Hello World";
Int32 index = 0;

Grouping

Group variable by their accessibility with a blank line. Use comments to separate variables with the same accessibility but part of the same context

Do not use #region.Â Classes should never be so large that you have the urge to introduce them, you should rather think of splitting classes into smaller pieces

// CORRECT
public int myVariable1 = 0;
public int myVariable2 = 0;
public int myVariable3 = 0;

protected int _myVariable = 0;

private int _myVariableHuman1 = 0;
private int _myVariableHuman2 = 0;
private int _myVariableHuman3 = 0;
// animals
private int _myVariableAnimal1 = 0;
private int _myVariableAnimal2 = 0;
private int _myVariableAnimal3 = 0;

// WRONG
#region
public int myVariable1 = 0;
public int myVariable2 = 0;
public int myVariable3 = 0;
protected int _myVariable = 0;
private int _myVariableHuman1 = 0;
private int _myVariableHuman2 = 0;
private int _myVariableHuman3 = 0;

private int _myVariableAnimal1 = 0;
private int _myVariableAnimal2 = 0;
private int _myVariableAnimal3 = 0;
#endregion

Variables

Variable names should speak for themselves and avoid abbreviations. (the only exception is the convention 'i' index abbreviation in loops). They should be declared in aÂ lower camel-case convention and not prefixed.

// CORRECT
public void Method(string stateName)
{
	GameStateMachine gameStateMachine = new GameStateMachine(stateName);
}

// WRONG
public void Method(string s, string _s)
{
	GameStateMachine gsm = new GameStateMachine(s);
	GameStateMachine _gsm = new GameStateMachine(_s);
}

Implicit Typing

Use implicit typing for local variables declarations.

// CORRECT
var text = "Hello World";
var number = 0;
var floating = .5f
var converted = number.ToString();
var lineConversion = Convert.ToInt32(Console.ReadLine());
var valueNumber = Object.Result();
var stringText = Object.Name();

Magic Numbers

Never use simple numbers in production code. When encountering hardcoded constants (Magic-Numbers) in code, create a constant with an explanatory name.

// CORRECT
const int MaxCountOfEntitites = 42;
for (int i = 0; i < MaxCountOfEntitites; i++)
{
}

// WRONG
for (int i = 0; i < 42; i++)
{
}

Unity inspector variables

When declaring a variable that is only needed to be ejected externally with objects via Unity editor, please make it private with a SerializeField attribute. This keeps the variables in the scope they intent to. Also use HideInInspector if we don't need the public variable to be accessed in the inspector. This will avoid unnecessary garbage in the Unity Editor and in the object external code access.

public class MyClass
{
	[HideInInspector] public int health;

	[SerializeField] private GameObject _externalObject;
}

Indentation

Code indentation

IndentÂ ONLYÂ withÂ tabsÂ andÂ notÂ with spaces. Your IDE should be configured to automatically write new lines with tabs with one level of indentation using 4 spaces.

Brackets

Always write an opening and a closing curly bracket. General practice should be having an opening and closing curly bracket in a separate line.
Exception is for one liner if cases

// CORRECT 
private void MyFunction()
{
    if(false) return;
    
	if(false)
	{
		return;
	}

	if (true)
	{
 		DoSomething();
	}
	else
	{
 		DoSomethingElse();
	}
	
}

// WRONG
private void MyFunction() {
	if(false)
		return;

	if (true){
 		DoSomething();
	}
	else{
 		DoSomethingElse();
	}

	if (true)
 		DoSomething();
	else
 		DoSomethingElse();
}

The only exception is when declaring properties with automatic or simple get/set configuration

// CORRECT 
public string Bar { get; set; }
public string Bar2
{
	get { return _anotherString; }
	set	{ _anotherString = value; }
}
public string Bar3
{
	get
	{
		return _anotherString;
	}
	set
	{
		_anotherString = value;
	}
}

// WRONG
public string Bar {	get	{ return _anotherString; } set { _anotherString = value; } }
public string Bar2
{
	get { DoSomething(); return _anotherString; }
	set	{ DoSomething(); _anotherString = value; }
}

Declarations

When declaring multiple variables, please do not align variable names with each other. AlsoÂ do not group declare multiple variables of the same type

// CORRECT
int myNumber;
int myOtherNumber;
int myAnotherNumber;
string myString;
bool myBoolean;
GameStateManager gameStateManager;

// WRONG
int myNumber, myOtherNumber, myAnotherNumber; 
string				myString;
bool				myBoolean;
GameStateManager	gameStateManager;

Switch-case

Please always indent the case statements of your switch-cases. Also, the usage of brackets within cases is enforced to enable scoping for the different cases, the only exception is when 2 cases have the same logic

// CORRECT
switch (myStatusVariable)
{
	case MyEnum.ValueOne:
	{
		DoSomething();
		break;
	}
	case MyEnum.ValueTwo:
	default:
	{
		DoSomethingElse();
		break;
	}
}

// WRONG
switch (myStatusVariable)
{
	case MyEnum.ValueOne:
		DoSomething();
	break;
	case MyEnum.ValueTwo:
	default:
 	DoSomethingElse();
}

Methods

In case of using methods with a very big number of parameters, it should be an indicator of poorly built code that will most likely violate any of the OOP or SOLID principles. But in the exception case of the method being to verbose and with a reasonable amount of parameters use the parameter name declaration scheme followed by the parameter reference:

QuantumEvent.Subscribe<EventOnProjectileFired>(
				listener: this,
				handler: HandleOnProjectileFired,
				once: false,
				onlyIfActiveAndEnabled: true);

A good rule of thumb practice is when a method declaration has more than 4 parameters, to convert it to a local private struct help organize better the data.

// WRONG
public void MethodDeclaration(string firstParamter, string secondParameter, string thirdParameter, 
                              string fourthParameter, string fifthParameter, string sixParameter)
{
}

// CORRECT
private struct DataParameter
{
    public string firstParamter;
    public string secondParameter;
    public string thirdParameter;
    public string fourthParameter;
    public string fifthParameter;
    public string sixParameter;
}

public void MethodDeclaration(DataParameter parameter)
{
}

Line Wrapping

Use UNIX-style line-wraps (LF). Please also set your IDE to display margin atÂ 128Â characters which should not be exceeded. This allows us to have multiple editor tabs open at the same time without having to scroll horizontally.

// CORRECT (variable and assignment separated by a new line to avoid an overlong line)
float myExtremeLongDescriptiveVariableName =
MyEnterpriseClassLevelFunctionFactoryDescriptor.
GetTotallyExtensiveFunctionResult(
reallyLongParamaterDescriptionContentObject);

// WRONG (class and method name separated by new line)
float myExtremeLongDescriptiveVariableName =
MyEnterpriseClassLevelFunctionFactoryDescriptor.
GetTotallyExtensiveFunctionResult
(reallyLongParamaterDescriptionContentObject);

// CORRECT (as many parameters on a single line as possible, overlong declarations wrapped to new line with indention)
public void GetTotallyExtensiveFunctionResult(string firstParamter, string secondParameter, 
                                              string thirdParameter, string fourthParameter)
{
}

// WRONG (every parameter on a single line)
public void GetTotallyExtensiveFunctionResult(string firstParamter,
                                              string secondParameter,
                                              string thirdParameter,
                                              string fourthParameter)
{
}

// CORRECT (as many conditions on a single line as possible, operatoron same line)
if (paramA == 1 && paramB == 2 && paramC == 3 &&
	paramD == 4 && paramE == 5)
{
}

// WRONG (conditions scattered over multiple lines, operator on next line, multiple indentions)
if (paramA == 1 && paramB == 2 && paramC == 3
	&& paramD == 4
	&& paramE == 5)
{
}

Spaces

After the keywords if, switch, for or while a space character is mandatory as it increases the readability. Also use spaces in expressions (x = 10) andÂ also add a space character after each comma (except at the end of the line)

// CORRECT
if (y == 0)
{
	x = 100;
}

// WRONG
if(y==0)
{
	x=100;
}

// CORRECT
private void DoSomethingWithParams(int firstParam, int secondParam)
{
}

// WRONG
private void AnUglyFunction (int uglyOne, int uglyTwo)
{
}
private void AnUglyFunction(int uglyOne,int uglyTwo)
{
}

Blank Lines

Use blank lines to separate blocks of code, methods and variables. Avoid unnecessary blank lines in the code when doesn't meet any criteria.

// CORRECT
private void DoSomething()
{
	Update();

	if (true)
	{
		DoSomethingElse();
	}
}

// WRONG
private void DoSomething()
{

	Update();



	if (true)
	{

		DoSomethingElse();
	}


	if (true)
	{
		DoSomethingElse();


	}


}

Comments

Important

Comments should be avoided in the middle the code. Generally the code should be self explanatory and if following correctly the coding guidelines, the should will be easily readable. For the special cases check the follow guidelines.

Indentation

Comments always have their separate lines and comment tokens are always followed by a white space

// CORRENT
//WRONG

// CORRENT
DoSomething

DoSomething; //WRONG

TODO Statements

When leaving a comment for something that is unclear, or that you have changed or simply a TODO, please also add your name. A task that is considered as done should never have any TODO statements remaining!

// TODO: migueltomas - I had to change this because it crashes.
