Blast Royale : Pull Request Guidelines

Created by Robert Gesicki, last modified on Feb 07, 2023

General Guideline

  • Pull requests reviewers should not have to wonder what it is that they are reviewing. Anybody should be able to go into any pending pull request, and understand what a PR is aiming to achieve, not just people who are actively involved in development with the PR owner.

  • Do NOT merge other peopleâ€™s PRs without explicit permission.

  • Check and fix ALL pipeline fails before merging. You should not be committing failing code into develop. We have a build and test pipeline for a reason.

    • If the pipeline is already failing from develop, then you can merge, but it would be preferable to fix those on the PR if itâ€™s a quick straightforward fix. Otherwise the relevant person should open up a separate PR to get the issues sorted.

Titles

Titles should be describing the feature concisely, or describing bug with valid Jira ticket code attached. Also, if a PR is not yet finished, or has known issues, title should have a [WIP] tag in the title.

Examples:

  • Weapon stat calculation improvements

  • New sound effects implemented for all weapons

  • [WIP] Updated reloading mechanics

  • [BR-567] Fix for players falling out of the map

  • [WIP, BR-898] Temporary workaround for player character model disappearing in match

Descriptions

Descriptions should have 2 sections: Summary, Tech Details

  • Summary section should give a brief description of what the feature is (or the fix), why it was done, and any additional notes that are relevant. This section should also have links to the relevant documents, data sheets, Jira tickets pertaining to the feature

  • Tech Details section should be a bullet point list of the only the KEY parts of the feature, and key scripts that were added/modified, ideally. Nothing crazy long, just a starting point for the reviewer to look into the implementation.

  • TODO is an optional section that can be included to let know the reviewer that there are issues currently on the branch that are yet to be fixed. All PRs with this section should be tagged as [WIP]. Do not merge branches that are work in progress - all issues should be resolved before merging.

Descriptions should also ideally reflect feedback that was implemented from reviews.
