Naming

Description

Prefix

Suffix

View

MonoBehaviours to handle the user input/output data to a single Presenter or Controller.

ViewModel

Interface to act between the View and the Logic. It binds the View‚Äôs input to its respective behaviour and binds the Logic‚Äôs output to its respective visualization.

Presenter

MonoBehaviours to be used to control IMGUI Views of the UI. Acts as a middle man between the view and the Logic.

Note: Use only for the UI. If needed to control 3D element views use a Controllers.

Logic/DataProvider

Interface to act over the model to manipulate the games main Data. It establishes a set of available operations with deterministic result. It only has internal access and should be decoupled from any View, Presenter, Controller, State or Service.

Controller

Similar to Presenters. MonoBehaviours to be used to control 3D view entities (GameObjects) in the Scene. Acts as a middle man between the 3D view and the Logic.

Note: Use only for the 3D GameObjects. If needed to control the UI element views use a Presenter.

Service

Interface that establishes a set of deterministic operations to support the execution of the game. It should be a stateless interface with public access in any part of the game.

System

Class object that is decoupled and isolated with any part of the game. It executes a set of instructions on update based on a set of ComponentData.

Note: Best use over an ECS architecture

Manager

Avoid this naming. It is a too generic definition to be understandable
