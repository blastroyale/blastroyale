Blast Royale : Release Cycles & Process

Created by Gabriel, last modified on Aug 01, 2022

The idea of this proposal is to mitigate release pain points.

Planning a Release Cycle:

  • Stakeholder define scope of the release and it‚Äôs objectives internally.

  • Stakeholder write down specifications for the release in an epic ticket in Jira and put this ticket in the release column (e.g what he wants, what needs to be achieved etc)

  • Planning meeting with the team and the stakeholder to make sure everyone understands the scope of the release and what‚Äôs need to be done, and if the timeline makes sense.

  • Start development.

Timeline:

Step 1 - Feature Freeze

  • No more features prioritized by devs or producers.

  • Release candidates made for all projects using gitflow & release branches (e.g release/1.2.3)

  • All teams communicate in #releases channel the release candidate versions

  • All release branches are deployed to staging environment

  • The bundle of versions are referred to a ‚Äúrelease candidate‚Äù

  • Stakeholders approve the release candidate feature set

  • QA Week begins on the RC

Step 2 - Release Drill & QA Week - ETA 5 days

  • Ensure staging is a replica of production¬† ¬† - Server versions, Client Version, Server Configuration, Marketplace Services.

  • Copy Playfab data, postgres & Moralis data from production to staging.

  • Release candidates are deployed to staging.

  • All external components are deployed (marketplace, game-logic service, game-client ready)

  • Engineering Pass to ensure nothing is missing

  • In parallel, QA Sanity Check test case runs. (Game & Marketplace)

  • Game is tested on both Android & IOS

  • In case of bugs, hotfix goes to release candidate and re-start.

Step 3 - Release - ETA 1 day

  • Place title into maintenance so players can‚Äôt play while the game is in maintenance. This should last 10/30min or 1h/4h for long maintenances

  • Deploy servers, marketplace

  • Whitelist QA to be able to log-in in GM account

  • Engineering pass checking metrics

  • QA does simplified sanity pass

  • Maintenance removed, game open for players

Step 4 - Post Release - ETA 3 days

  • Server metrics actively monitored

  • Game analytics actively monitored

  • Community actively monitored

  • Devs focused on finding and solving pain points uncovered during release, staging or known bugs that should be addressed.

QA Sanity Check:

  • QA checks technical checklist (Release Checklist - Blast Royale - Confluence (atlassian.net))

  • QA Logs in a player account and play the game as a random player.

  • QA runs the marketplace test cases Marketplace QA Test-Cases
